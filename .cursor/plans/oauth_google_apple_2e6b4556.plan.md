# OAuth Google / Apple для новых пользователей + Telegram без удаления

## Цель продукта

- **Новые пользователи** проходят регистрацию/вход через **Google или Apple** (основной планируемый веб-онбординг).
- **Telegram** остаётся: бот по-прежнему может создавать и обслуживать пользователей по `telegram_id`; старые JWT и ссылки не ломаются намеренно.

Это меняет акцент по сравнению с вариантом «OAuth только для привязки к уже существующему telegram-аккаунту»: при первом успешном OAuth **нужно создать запись пользователя и цепочку provision сервера**, если её ещё нет (аналог того, что сейчас делает бот при регистрации).

## Как устроено сейчас (релевантно)

- [`users_subscription`](db/models.py): PK **`telegram_id`**; вся логика ([`UserControl`](methods/manager_users.py), [`subscription`](routers/subscription.py)) и JWT завязаны на поле `telegram_id` в payload.
- JWT: `jwt.encode({"telegram_id": ...}, ...)`.

## Проблема

Пользователь **без Telegram** не имеет настоящего `telegram_id`. Нужен единый **внутренний идентификатор аккаунта**, который:

- попадает в JWT (можно сохранить имя ключа `telegram_id` для обратной совместимости с клиентами, если значение — ваш внутренний bigint; либо ввести `user_id` в payload и обновить все декодеры — второй путь чище, но больше правок).

## Варианты модели данных (выбрать один до реализации)

**Вариант A — минимальная смена смысла поля**

- Завести **суррогатный числовой id** (например `user_id` BIGINT PK), `telegram_id` сделать **NULLABLE UNIQUE** (реальный Telegram, если есть).
- Для OAuth-only пользователей `telegram_id IS NULL`; в JWT передавать `user_id` (потребуется **обновить** все места с `payload['telegram_id']` и репозитории — на `get_by_user_id` или единый резолвер).

**Вариант B — сохранить один ключ в JWT**

- Оставить одну колонку «как сейчас», но для не-Telegram выдавать **синтетический** `telegram_id` из **зарезервированного диапазона** (например только отрицательные BIGINT или диапазон вне реальных tg id), чтобы не пересечься с настоящими Telegram.
- Плюс: меньше переименований в JWT. Минусы: семантика «telegram_id» становится вводящей в заблуждение; нужно жёстко документировать диапазон.

**Вариант C — полный рефакторинг FK на `user_id`**

- Новый PK `user_id`, `telegram_id` опционален; все FK (`sale_invoices`, `invited`, и т.д.) перевести на `user_id`. Самый чистый долгосрочно, самый трудозатратный.

Рекомендация для плана: зафиксировать выбор между **A** и **B** на созвоне/ревью; от этого зависят миграции и объём правок в [`UsersRepository`](db/repository/users.py) и JWT.

## Таблица OAuth

Независимо от варианта выше:

- `user_oauth_identity`: связь `(provider, subject)` → аккаунт (FK на PK пользователя), уникальный индекс по `(provider, subject)`.
- При первом входе: после валидации id_token создаётся пользователь (и при необходимости — сервер/ссылка через существующие `UserControl`/`ServersRepository` по той же бизнес-логике, что и для бота).

## Потоки

### Новый пользователь (Google/Apple)

1. Старт OAuth → callback с `state`.
2. Проверка токена провайдера, извлечение стабильного `sub`.
3. Если записи нет — **создание пользователя** и provision VPN; запись в `user_oauth_identity`.
4. Выдача JWT (согласованного с выбранной моделью id).
5. Редирект на [`/sub/home?token=...`](routers/subscription.py) или отдельный «кабинет».

### Существующий пользователь Telegram

- Без изменений: бот, старые ссылки с `jwt`/`token`.

### Слияние аккаунтов (не обязательно в первой версии)

- Если один человек сначала OAuth, потом Telegram — нужна отдельная политика (запрет двух аккаунтов, или явная «привязка»). Вынести в follow-up, иначе раздувает scope.

## Изменения в коде (высокий уровень)

| Область | Действие |
|---------|----------|
| БД | Миграция по выбранному варианту A/B/C + `user_oauth_identity` |
| JWT | Единая функция выпуска токена и единая функция «user из query token» (рефактор [`get_current_user`](methods/manager_users.py) и дубли в subscription) |
| Роуты | Новый blueprint OAuth (start/callback; для Apple — учёт `response_mode=form_post`) |
| Онбординг | Повторное использование логики создания пользователя на сервере (как бот), без дублирования бизнес-правил |
| Конфиг | Секции Google/Apple, redirect URIs |

## Риски

- **Оплата для OAuth-only**: если сейчас оплата идёт только из бота, для новых пользователей понадобится **веб-оплата** или редирект в бота — это отдельное продуктовое решение (в план не включать без уточнения).
- **Redirect URI / домены** для Google и Apple.
- **Apple**: веб-токены и нативные отличаются; заранее определить платформу (web-only vs iOS+web).

## Что убрано из первоначального плана

- Обязательная привязка через одноразовый код **из бота** как единственный путь — не соответствует цели «новые пользователи через OAuth». Код из бота остаётся **опциональным** для слияния аккаунтов (todo `optional-link-telegram`).
